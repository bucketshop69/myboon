# Predict P0 Release Readiness QA Guide

Issue: #158 P0 Predict release readiness

## Purpose

This guide covers the manual checks required before shipping the Predict P0 release. The focus is account setup, wallet/session state, trading actions, portfolio cash accounting, and the locked release scope.

## Environment And Data Notes

- Test on a clean install and on an upgraded install with an existing Predict account.
- Run at least one pass on iOS or Android Expo, and one pass on web if web is part of the release candidate.
- Use a funded test wallet with a small USDC balance that can safely place live Polymarket orders. Keep trade sizes near the minimum supported size.
- Use a second wallet with no Predict account and no pUSD/USDC balance.
- Use a wallet that already has active positions, at least one open order, and at least one redeemable resolved position. If this is not available, record the missing fixture and block the relevant case until Product/Eng provides one.
- Use a market with enough liquidity for a small FOK buy and another market/outcome/price combination that reliably returns a not-filled FOK result.
- Use a position with a cash-out quote that can be filled and one position or price condition expected to return no fill.
- Network shaping is required for delayed deposit, stale refresh, and interrupted setup checks. Use simulator network controls, proxy throttling, or API stubs when live timing is impractical.
- Capture screenshots or video for every failure and include wallet address, route, market slug, action, timestamp, app build, API base URL, and observed response/error.

## Release Gates

- All P0 cases below must pass on the primary mobile target.
- No stale wallet, stale cash, or stale order state may remain visible after disconnect, wallet switch, refresh, or session expiry.
- Failed, not-filled, delayed, and expired states must be explicit and recoverable.
- Swap and Perps preview surfaces must stay preview-only for this release.

## Manual Test Cases

| ID | Area | Preconditions | Steps | Expected result |
| --- | --- | --- | --- | --- |
| PRED-001 | Predict sign-in wording | App is signed out and no wallet is connected. | Open `Predict Profile` or any Predict gated action. Review the empty state and primary CTA. Tap `Sign In`. | Copy says `Sign in to use Predict` or equivalent Predict-specific wording. CTA opens wallet sign-in/setup, not a trade. No language implies seed phrase creation, gas, or a paid transaction. |
| PRED-002 | Predict setup wording after wallet connect | Solana wallet is connected, Predict account is not enabled. | Open Predict profile. Tap `Open Prediction Account`. On native, inspect the confirmation alert. | Alert title is `Connect Predict Account`. Body explains one signature to restore or set up the prediction account, verifies ownership, has no transaction, no extra seed phrases/wallets, and allows deposit/picks without gas. `Cancel` dismisses without enabling; `Continue` starts setup. |
| PRED-003 | Privy email/passkey wallet preparing | Fresh device/browser, no existing Privy session. | Start Predict sign-in with email/passkey. Complete OTP/passkey prompts. Keep the app foregrounded during wallet preparation. | UI shows a preparing/loading state until Privy wallet/session is usable. The user is not dropped into a blank profile or trade screen. When complete, profile shows connected identity and the next Predict setup action. |
| PRED-004 | Privy preparing interruption | Same as PRED-003. | Start email/passkey setup, background the app or briefly disable network before wallet preparation completes, then return/restore network. | Preparing state resumes or shows a clear recoverable error. Retrying does not create duplicate visible accounts and does not show stale wallet addresses. |
| PRED-005 | Wallet disconnect clears stale state | Predict account is enabled and portfolio data is visible. | Open wallet/settings, disconnect the wallet. Return to Predict profile and a market detail screen. | Profile and detail screens stop showing prior wallet cash, positions, open orders, and deposit/withdraw controls. CTA returns to sign-in/connect state. Buy/cash-out/redeem actions are blocked. |
| PRED-006 | Wallet switch clears stale state | Wallet A has Predict data. Wallet B is connected with different/no Predict data. | While on Predict profile, switch from Wallet A to Wallet B. Repeat from a market detail screen with the bet slip open. | Every wallet-bound field refreshes for Wallet B. Wallet A cash, positions, open orders, pending actions, and deposit address do not remain visible. Any open bet slip revalidates available cash/session before submit. |
| PRED-007 | Setup interrupted by wallet switch | Wallet A connected, Predict setup not enabled. Wallet B available. | Start `Open Prediction Account` for Wallet A. Before or during signing, switch to Wallet B or disconnect. Complete/cancel the original prompt if it remains open. | Setup must not enable or display Wallet A account under Wallet B. App shows a clear retry/connect state for the current wallet. No deposit address, polygon address, or trading address from Wallet A leaks into Wallet B UI. |
| PRED-008 | Buy FOK not filled | Predict account enabled, available cash >= minimum order. Choose a market/outcome/amount expected to fail FOK fill. | Open market detail. Select outcome. Enter amount. Submit the buy. | Submit uses FOK behavior. User sees `Not filled. Price or liquidity changed. Try a smaller amount or refresh the market.` or equivalent. No open order or position is added. Available cash is unchanged after refresh. |
| PRED-009 | Buy FOK filled | Predict account enabled, available cash >= minimum order, liquid market selected. | Open market detail. Select outcome. Enter a small amount. Submit the buy. Refresh profile/detail after success. | Submit shows progress and then success/recovery state. Position or activity row appears with the correct market/outcome. Available cash decreases by executed cost. Reserved/open-order cash is not created for a fully filled FOK order. |
| PRED-010 | Buy guardrails | Predict account enabled with low or known cash balance. | Try amount 0, below minimum, above available cash, and with stale price after waiting for odds to move. | Confirm button blocks invalid amounts with clear copy such as `Enter an amount`, `Minimum order size`, or `Not enough cash`. Stale price/liquidity failures return a not-filled/retry message, not a silent failure. |
| PRED-011 | Cancel requested | Predict account has at least one open order. | Open profile or detail `Your Picks`. Tap the open order, request cancel, and keep the app foregrounded. | Row enters a cancelling/requested state and disables duplicate cancel taps. After refresh, the open order disappears or updates to canceled. Reserved cash is released into available cash. |
| PRED-012 | Cancel failure | Predict account has an open order. API/network can be forced to fail cancel. | Request cancel while offline or with cancel endpoint returning error. | User sees a failure message. Row exits busy state and remains actionable if the order is still open. Reserved cash is not released unless backend confirms cancellation. Retry can be attempted. |
| PRED-013 | Cash-out FOK no-fill | Account has an active position with cash-out available but expected no FOK fill. | Open position action. Choose `Cash out`. Confirm cash-out. | User sees a no-fill/retry message. Position remains active. Cash and position value do not change after refresh. No duplicate pending cash-out row remains. |
| PRED-014 | Cash-out FOK filled | Account has an active position with a fillable cash-out quote. | Open position action. Choose `Cash out`. Confirm and refresh profile/detail. | Cash-out submits as FOK and completes. Position size/value decreases or closes. Available cash increases by proceeds. Activity reflects the updated position state. |
| PRED-015 | Redeem collecting | Account has at least one redeemable resolved winning position. | Open profile or detail `Your Picks`. Tap redeem/collect on the redeemable row. | Row enters collecting/busy state. Duplicate taps are blocked. On success, redeemable row disappears or moves to collected/closed state and cash/profit updates after refresh. |
| PRED-016 | Redeem failure | Redeemable row exists. API/network can be forced to fail redeem. | Trigger redeem while endpoint fails or wallet/session is invalid. | User sees a clear failure such as `Could not redeem this payout. Try again in a moment.` or the mapped server error. Row exits busy state, remains redeemable, and retry works after recovery. |
| PRED-017 | Redeem session expired | Predict account enabled, then expire/revoke session before redeem. | Open profile/detail and attempt redeem. | App shows `Session expired - tap to reconnect` or equivalent reconnect path. Redeem is blocked until reconnect succeeds. After reconnect, portfolio reloads and redeem can be retried. |
| PRED-018 | Deposit delayed tracker | Predict account enabled. Deposit modal available. | Open `Deposit`, copy a supported address, but do not send funds. Keep modal open past the delayed threshold or simulate no bridge detection. Tap refresh. | Tracker persists after copy and shows waiting first, then `Deposit delayed` with guidance. Refresh remains available. Closing/reopening does not falsely report success. |
| PRED-019 | Deposit persistent tracker success | Predict account enabled, small supported deposit sent to copied address. | Open `Deposit`, copy Solana/EVM address, send funds, keep modal open, and observe polling. | Tracker shows waiting/detected/bridging/finalizing as available. When balance increases, it shows `Funds available`, calls portfolio refresh, and the cash balance reflects the deposit. |
| PRED-020 | Deposit tracker persistence across navigation | Deposit tracker active. | Copy an address, close the modal, navigate away/back, reopen deposit. Repeat with app background/foreground. | No stale false-positive success is shown. If tracker is designed to reset on close, it resets cleanly; if designed to persist during the session, it keeps the same address/status. In both cases, cash balance remains source-of-truth after refresh. |
| PRED-021 | Withdraw submitted | Predict account enabled with available cash >= $1. | Open `Withdraw`, enter valid amount and Solana address, review, submit. | Confirm screen shows amount, source `Polymarket Safe`, destination, and `Free (Builder)`. Success state says `Withdraw submitted!`, includes tx hash when returned, and refreshes cash. |
| PRED-022 | Withdraw validation/failure | Predict account enabled. | Try amount below $1, above cash, invalid Solana address, and an API/network failure after submit. | Invalid input keeps `Review Withdraw` disabled and shows address error when relevant. API failure shows `Withdraw failed`, preserves retry path, and does not reduce cash unless backend confirms submission. |
| PRED-023 | Available vs reserved cash | Account has available cash and at least one open order. | Record cash before open order. Place or use an open order. Compare profile cash, active picks, order cost, and market detail guardrails. Cancel the order and refresh. | Available cash excludes reserved order cost. Active/open order value is visible separately in picks/activity. Buy `Max` and insufficient-cash checks use available cash, not total portfolio value. After cancel, available cash increases by the released reserved amount. |
| PRED-024 | Stale refresh behavior | Account has Predict data. | Pull-to-refresh profile. Let app sit for several minutes. Background/foreground. Open market detail and wait for polling. Force one portfolio/order API failure. | Freshness/loading state is visible during refresh. Data updates when focused and does not regress to older wallet data. On failure, existing data remains visible with recoverable error/freshness copy, and subsequent refresh replaces stale values. |
| PRED-025 | Stale market price before submit | Market detail open with selected outcome and amount. | Wait for odds/orderbook to change, or simulate live price moving after quote display. Submit. | App either refreshes quote before submit or returns clear not-filled/stale price feedback. Filled orders use the latest executable price shown in activity/position. |
| PRED-026 | Live executable odds | Market detail open on a liquid binary and sports market. | Compare the visible outcome percentages against the Book tab/best ask from the proxied orderbook. Have another account take liquidity or wait for the book to move, then observe the open screen for at least one polling interval. | Visible buy odds track the nearest executable ask from the live orderbook, not stale details API percentages. The bet slip quote updates from the same source and submit re-fetches the book before placing FOK. |
| PRED-027 | Release scope lock: Swap preview | Home or market preview includes Swap entry point. | Tap Swap preview and any related preview action. | Swap remains preview-only for this release. It must not expose a production swap execution path unless explicitly in scope. Any CTA should be disabled, placeholder, or clearly route outside Predict P0 acceptance. |
| PRED-028 | Release scope lock: Perps preview | Home or market preview includes Perps entry point. | Tap Perps preview and any related preview action. | Perps remains preview-only for this release. It must not expose new perps trading execution as part of Predict P0. Existing preview/list navigation must not break Predict state or wallet session. |

## Final Readiness Checklist

- [ ] Sign-in/setup copy approved by Product.
- [ ] Wallet disconnect, wallet switch, and setup interruption produce no stale state.
- [ ] Buy, cash-out, cancel, redeem, deposit, and withdraw each have success and failure evidence.
- [ ] Available cash, reserved cash, positions, open orders, and ready-to-collect values reconcile after refresh.
- [ ] Visible market percentages match live executable orderbook ask prices.
- [ ] Session expiry and reconnect path tested on profile and detail actions.
- [ ] Delayed deposit tracker tested for both waiting/delayed and funds-available states.
- [ ] Swap and Perps preview scope lock verified.
- [ ] Known fixture gaps and waived cases are documented with owner and follow-up issue.
